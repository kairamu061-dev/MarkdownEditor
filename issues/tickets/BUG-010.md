# BUG-010: 引用行の改行→Backspace の直後に日本語 IME 変換確定すると文字化けする

- **State**: Open
- **Reported**: 2026-07-20
- **Area**: editor/core（@codemirror/lang-markdown・IME composition）

## Summary

引用行（`>`）を自動継続で改行し、増えた引用行を Backspace で削除した直後、その位置で日本語 IME 変換を確定すると、確定文字がローマ字断片＋ひらがな重複に化け、削除したはずの `>` 引用マークが復活する。編集画面の表示と実ファイル内容も食い違う。

## Reproduction

1. ノートに `> quote` と入力
2. Enter を押す（`> ` が自動継続して空の引用行ができる）
3. Backspace を押す（増えた引用行を削除）
4. IME を ON にして、その位置で `にほんご` などを変換・確定

観察: 確定文字が `ｎｎほんごほんごほんｇほｎほｈ` のようにローマ字断片＋ひらがな重複に化け、`>` 引用マークが復活する。

### コントロール付き A/B（2026-07-20 実機・自動操作）

| ケース | 確定後のディスク内容 |
|---|---|
| A: 通常行での日本語入力（対照） | `にほんご` ✅ 正常 |
| B: 引用＋Backspace の直後 | `> quote` ＋ 空行 ＋ `> ｎｎほんごほんごほんｇほｎほｈ` ❌ 崩壊 |

違いは「引用行を作って改行→Backspace した直後か」だけ。壊れ方が 2 回とも寸分違わず同一（`ｎｎほんごほんごほんｇほｎほｈ`）で、決定的なバグと判断。

## Expected

引用＋Backspace の直後であっても、日本語 IME の変換確定は通常行と同じく正常に反映される（`にほんご`）。`>` マークが勝手に復活しない。

## Cause

（調査中）2026-07-21 の静的調査で、リポジトリ独自コードではなく CodeMirror（`@codemirror/view`）の
**「変更直後に IME composition が始まると DOM が壊れる」既知バグ群**に属する可能性が高いと判明。

### コードパスの確認

`node_modules/@codemirror/lang-markdown@6.5.1` の `deleteMarkupBackward`（`dist/index.js:348`）を確認:

1. `> quote` の行末で Enter → `insertNewlineContinueMarkup` が `\n> ` を挿入し、空の引用継続行ができる。
2. Backspace → `deleteMarkupBackward` が発火。空引用行では `/^[\s>]*$/` 条件が真になり、`> ` マークを削除して空行にする（`dist/index.js:366-378`）。ここまでは単体では正常（BUG-008 の範囲）。
3. その空行位置で IME composition を開始・確定 → 崩壊。

つまり「**構造を変えるトランザクション（`> ` 削除）→ 直後に IME composition 開始**」という並びが引き金。リポジトリ側（`src/editor/`）に引用/IME の独自処理は無く、`live-preview.ts` も blockquote を特別扱いしていない。

### IME 入力経路の特定（2026-07-21 修正）

当初は WebView2 の **EditContext API** 経路を疑ったが、**それは誤り**だった。
`@codemirror/view@6.43.6` の `dist/index.js:7132` を確認すると、EditContext は
`browser.android`（= `navigator.userAgent` に `Android` を含む）が真のときにしか生成されない:

```js
if (window.EditContext && browser.android && view.constructor.EDIT_CONTEXT !== false && ...)
```

WebView2 on Windows 11 のデスクトップは userAgent が `Windows NT ... Edg/...` で **Android ではない**ため、
**EditContext は使われない**。実際の IME 入力は CodeMirror の**従来型（classic）composition 経路**を通る
（`compositionstart`/`compositionend` と `DOMObserver` による DOM ミューテーション読み取り→`compositionend` で差分再構築）。

### 上流の証拠（`@codemirror/view` changelog）

classic composition 経路にも「変更直後に composition が始まると DOM が壊れる」系のバグが繰り返し修正されている:

- 6.39.16「changing decorations **right before a composition** could end up corrupting the visible DOM」← 本件と同型
- 6.14.1「compositions disrupted because their surrounding DOM was **repurposed**」
- 6.11.1 / 6.13.0 / 6.15.x: composition が改行や周辺 DOM を壊す系の修正

本件はまさに「構造変更（`deleteMarkupBackward` による `> ` 削除）の直後に composition 開始」というパターン。
`deleteMarkupBackward` の後、シンタックスハイライト/記法非表示のデコレーション再計算と DOM 差し替えが起き、
その直後に始まった composition が `DOMObserver` の再構築とズレて崩壊していると推測される。
（EditContext は無関係。上流の該当クラスの修正は既に 6.43.x に入っているため、単純な更新では直らない可能性がある。）

### 適用済み（2026-07-21）— 候補 A: CodeMirror パッケージの最新化

- `@codemirror/view` 6.43.5 → **6.43.6**
- `@codemirror/state` 6.7.0 → **6.7.1**
- 他（`codemirror` / `lang-markdown` / `language-data` / `commands` / `language` / `@lezer/highlight`）は既に最新。
- `npm run build`（tsc + vite）は成功。ただし **Linux では WebView2/日本語 IME を再現できず、本バグの解消は未検証**。

**確度が低い理由**（過信しないこと）:
- 6.43.6 の修正は「**zero-length** content updates での tile tree 破損」で、本件（`> ` の非空削除＋非空の IME 挿入）とは条件が一致しない。
- 本件と同型の「decoration change **right before a composition**」修正（6.39.16）は **既に 6.43.5 に入っており**、それでも再現していた。
- → **更新だけで直る保証はない**。Windows 実機で再検証し、直らなければ候補 B へ。

### 候補 B（A で直らない場合）

1. 最小再現（引用 Enter→Backspace→IME 確定）を作り、Windows 実機で切り分け。
   - 切り分けのため、`live-preview`/シンタックスハイライトの**デコレーションを外した素の CodeMirror**でも再現するか確認する。
     再現するなら CodeMirror の classic composition のバグ、しないなら本リポジトリのデコレーション再計算が誘因、と切り分けられる。
2. classic composition 経路の問題として CodeMirror 本家（`@codemirror/view`）へ最小再現付きで issue 起票。
   WebView2 ランタイムのバージョンも併記（WebView2 固有の composition 挙動の可能性）。
3. 回避策の検討: `deleteMarkupBackward` 直後に composition が始まる並びを避ける方向（例: Backspace 直後の状態を安定させる／
   問題が続く場合は該当行のデコレーションを一時抑制する）。いずれも Windows 実機での検証必須。

※ EditContext 経路は本環境では未使用のため、EditContext 無効化系の回避策は不要（上記「IME 入力経路の特定」参照）。
